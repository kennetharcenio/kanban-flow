● Great, here are three approaches for adopting TDD with retroactive
  comprehensive coverage:

  ---
  Approach A: Bottom-Up (Recommended)

  Start with models/utilities → services → components → dialogs → CI

  1. Build test infrastructure first (helpers, factories, mocks)
  2. Write tests for models and pure functions (easiest, no Angular deps)
  3. Move to core services (event-store, auth, sync)
  4. Then components (board, board-list)
  5. Finally dialogs and shared components
  6. Set up CI last once everything passes

  Pros: Each layer builds on tested foundations. Factories created for services
  get reused in component tests. Natural dependency order.
  Cons: Takes longer before you see component-level test coverage.

  Approach B: Risk-First

  Start with the most broken/critical code from the code review

  1. Build minimal test infrastructure
  2. Write tests exposing the critical bugs (columnId mismatch, auth bypass,
  cascade delete)
  3. Fix bugs with TDD
  4. Expand outward to remaining services and components
  5. CI setup

  Pros: Immediate value — bugs get caught and fixed first. Motivating.
  Cons: Test infrastructure gets built ad-hoc, may need refactoring. Jump
  between layers.

  Approach C: Feature-Slice Vertical

  Test one complete feature at a time (e.g., all of "board" — model + service + 
  component + dialog)

  1. Build test infrastructure
  2. Pick a feature slice (e.g., board management)
  3. Write all tests for that slice top-to-bottom
  4. Repeat for next slice
  5. CI setup

  Pros: Complete coverage per feature. Easy to track progress.
  Cons: Shared services (auth, event-store) get tested multiple times or
  awkwardly split across slices.

  ---
  My recommendation: Approach A (Bottom-Up). Since you want comprehensive 80%+
  coverage and full infrastructure, building layer by layer avoids rework. The
  test utilities and mock factories you build for services naturally feed into
  component tests. And having CI last means it gets set up when the full suite
  is ready.
